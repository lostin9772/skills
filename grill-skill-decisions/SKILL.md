# Grill Skill Decisions

## 核心目标

在用户写 `Codex Skill` 或做重要方案决策前，先通过一轮深入追问把隐含假设、触发条件、边界、失败模式和验证标准澄清出来。

这个 skill 的产物不是马上写文件，而是让 agent 和用户形成足够清晰的共同理解。只有当关键决策已经收敛时，才进入 `SKILL.md`、`agents/openai.yaml`、方案、spec 或实施计划的生成。

## 基本规则

- 一次只问一个问题，并等待用户回答。
- 能通过读取代码库、已有 `SKILL.md`、项目文档、日志、配置或上下文确认的信息，先读取，不要直接问用户。
- 每个问题都必须说明“为什么问”，并给出 agent 的推荐答案或候选选项。
- 不要一次性列出问题清单。
- 不要把用户的模糊表达当作已确认结论。
- 不要过早生成最终 skill、方案或代码。
- 除非用户明确要求写入文件，否则只在对话中推进。
- 技术参数、命令、编译选项、工具名、API 名、错误码、文件名、目录名和触发关键词保持英文原文。

## 启动判断

先判断用户是否已经给出明确对象。

如果对象不明确，只问一个问题：

```text
问题：这次你想澄清的对象是什么？是准备写一个 Codex Skill、设计一个工作流、做一个架构决策、整理一个操作规范，还是别的方案？

为什么问：不同对象需要追问的决策树不一样。先确定对象，后面的追问才能集中。

我的建议：如果你还没完全想清楚，可以先用一句话描述“我希望 agent 以后在什么场景下做得更好”。
```

如果对象明确，直接进入决策树追问。

## 问题格式

每次提问使用这个格式：

```text
问题：...

为什么问：...

我的建议：...

可选答案：
A. ...
B. ...
C. ...
```

如果开放式问题更合适，可以省略 `可选答案`，但仍然必须给出 `我的建议`。

## 决策树

按顺序推进。当前层级没有收敛前，不进入下一层。

### 1. 目标与失败模式

先确认为什么需要这个 skill 或方案。

必须弄清：

- 用户反复遇到的真实问题是什么。
- 普通 agent 默认容易在哪里做错。
- 这个 skill 或方案要防止哪类失败。
- 这是一次性需求，还是会跨项目反复出现。
- 成功后用户希望减少哪类沟通成本。

优先问题示例：

```text
问题：这个 skill 最核心要防止 agent 出现哪一种错误？

为什么问：skill 应围绕可重复出现的失败模式设计，而不是写成泛泛说明。

我的建议：优先选择“agent 默认会遗漏、误判或过早行动”的失败点，因为这类问题最值得写进 skill。
```

### 2. 触发条件与非触发条件

确认什么时候应该使用，什么时候不应该使用。

必须弄清：

- 用户会用什么关键词或任务描述触发它。
- 是否包含 `skill`、`SKILL.md`、`agents/openai.yaml`、`scripts/`、`grill-me`、`decision`、`workflow` 等关键词。
- agent 应该主动判断触发，还是只在用户显式要求时触发。
- 哪些相似任务不应该触发。
- 如果和其他 skill 重叠，哪个优先。

### 3. 使用环境与权限边界

确认执行时能看什么、能做什么、不能做什么。

必须弄清：

- 默认运行环境，例如 Windows、PowerShell、VS2019、CMake、GitHub、TortoiseGit 或某个固定 repo。
- 是否需要读取本地文件、代码、日志、截图、网页或命令输出。
- 是否允许创建、修改、删除文件。
- 是否允许运行命令、联网、安装依赖或启动服务。
- 哪些动作必须先向用户确认。

### 4. 工作流与追问策略

把 agent 的执行过程拆成可复用步骤。

必须弄清：

- 第一步必须做什么。
- 什么信息必须先确认。
- 遇到用户回答模糊时怎么继续追问。
- 遇到矛盾约束时怎么处理。
- 何时给出 2-3 个方案和推荐项。
- 何时停止追问并进入最终输出。

### 5. 输出物

确认最终交付的形态。

必须弄清：

- 最终是生成 `SKILL.md`、`agents/openai.yaml`、设计文档、决策摘要、spec、实施计划，还是只形成对话共识。
- 输出是否需要直接可复制使用。
- 是否需要包含已确认决策、默认假设和未决问题。
- 是否需要保留 Markdown 结构、代码块、命令、路径和字段名。
- 是否允许创建 `scripts/`，以及脚本是否真的用于重复、机械、容易出错的步骤。

### 6. Skill 文件结构

当目标是写 `Codex Skill` 时，必须额外确认文件结构。

默认只允许：

```text
skill-name/
├── SKILL.md
└── agents/
    └── openai.yaml
```

只有当脚本能明显减少重复、机械、容易出错的操作时，才允许增加：

```text
skill-name/
└── scripts/
```

不要创建 `references/`、`assets/`、`README.md`、`CHANGELOG.md`、`INSTALLATION_GUIDE.md` 或其他额外文档，除非用户明确改变要求。

### 7. 质量标准与验证

确认怎样算做对。

必须弄清：

- `frontmatter.name` 是否是英文 `kebab-case`。
- `frontmatter.description` 是否描述触发条件，而不是复述工作流。
- `description` 是否保留关键英文触发词。
- `SKILL.md` 正文是否符合用户要求的语言。
- 是否需要检查 YAML、目录结构、禁止目录和占位符。
- 是否需要运行 `quick_validate.py` 或等价格式检查。
- 是否需要用真实任务或子 agent 做压力测试。

## 写 Skill 前的收敛条件

只有能够清楚回答以下问题时，才可以开始生成最终 skill：

- 这个 skill 要解决什么重复问题。
- 谁会在什么场景下使用它。
- 它应该被哪些关键词或任务触发。
- 它不应该在哪些相似场景触发。
- 它必须遵守哪些工作流。
- 它必须避免哪些失败。
- 它的最终输出物是什么。
- 它允许创建哪些文件和目录。
- 它怎样验证格式和行为正确。
- 还有哪些默认假设或未决问题。

如果任一项仍然含糊，继续追问，不要生成最终文件。

## 最终输出规则

当用户要求生成 `Codex Skill` 时，输出或创建以下内容：

1. skill 名称。
2. skill 路径。
3. `SKILL.md`。
4. `agents/openai.yaml`。
5. 可选的 `scripts/`，仅在确实需要时创建。
6. 基本格式检查结果。

不要创建用户未允许的目录或文档。

## 常见错误

| 错误 | 正确做法 |
|---|---|
| 一次问很多问题 | 一次只问一个最关键问题 |
| 用户说“你看着办”就继续 | 给出推荐答案，让用户确认或修正 |
| 还没确认触发条件就写 `SKILL.md` | 先确认使用场景、触发词和非触发条件 |
| 把原始提示词整段堆进去 | 整理为可复用、可执行的工作流 |
| 为说明文字创建 `scripts/` | 只有重复、机械、容易出错的步骤才脚本化 |
| 创建 `references/` 或 `assets/` | 除非用户明确允许，否则不要创建 |
| `description` 写成流程摘要 | `description` 只写触发条件和适用场景 |
| 最后不验证 | 至少检查 frontmatter、目录结构、占位符和禁止目录 |

## 红旗信号

出现以下情况时，停止生成最终文件，继续澄清：

- 用户说“随便”“都可以”“你决定”。
- 目标同时覆盖太多互不相关的场景。
- 没有明确失败案例。
- 没有明确触发条件。
- 没有非目标或非触发条件。
- 没有验证方式。
- 和已有 skill 职责明显重叠。
- 输出物不清楚。
- 需要写文件但目标路径未确认。
- 需要运行命令但权限边界未确认。

## 最小示例

用户：

```text
我想写一个 skill，帮我以后打包 C++ 静态库。
```

agent：

```text
问题：这个 skill 最核心要防止哪类失败？是编译参数不一致、导出的 include/lib 不完整、consumer project 无法链接，还是测试流程缺失？

为什么问：skill 应该围绕最容易反复出错的失败模式设计，否则会变成泛泛的打包说明。

我的建议：优先把 consumer project 能否在目标工具链下独立编译、链接、运行作为核心成功标准，因为它能反向覆盖 include、lib、runtime library、C++ standard 和 debug symbols 等问题。

可选答案：
A. 以 consumer project 验证为核心
B. 以 package 目录结构为核心
C. 以编译参数一致性为核心
```

然后等待用户回答，再继续下一个问题。
